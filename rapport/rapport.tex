\chapter{Introduction}

Je commencerais par évoquer les concepts sous-jacents à l'apparition
du cloud. Il s'agit de "poser" les notions manipulées par le travail
qui va suivre.

Comme souvent, les choses se construisent au fur et à mesure.

\section{Evolutions \ldots vers l'infrastructure cloud}
\subsection{Abstraction}
Vouloir abstraire la machine sur laquelle on veut faire un exécuter un
programme est un souhait qui apparaît avec les premiers systèmes
d'exploitation.

Bien avant le cloud, au milieu des années 60, Multics (Multiplexed
Information and Computing Service) est un système d'exploitation conçu
avec une vision de "Computer Utility" (service informatique public).

Pour ses créateurs (MIT, Bell Labs et GE), la puissance de calcul doit
être "accédée'' comme l'électricité : vous branchez une console dans
le mur, et vous avez accès à une ressource partagée, sans savoir où se
trouve la machine.

Multics avait la philosophie du Cloud (l'informatique comme service
public partagé) et l'architecture du Cloud (modularité, isolation,
persistance), mais il lui manquait la puissance des machines et la
couche réseau mondiale pour aboutir.

\subsection{Accés distant}
Les machines ont toujours été développées avec l'idée d'être accédées à distance.

G.Stibitz développe le premier additionneur electromécanique
sur 2 bits en 1937. \cite{WikipediaAdder2026}

% ref : \url{https://en.wikipedia.org/wiki/Adder_(electronics)}

Immédiatement, il complexifie son calculateur et imagine un accès
distant. Lors d'une démonstration à la réunion de l'American
Mathematical Society au Dartmouth College en septembre 1940, Stibitz
utilisa un télétype modifié pour envoyer des commandes par lignes
télégraphiques au Complex Number Calculator situé à New York. \cite{Stibitz2012}

% ref : https://history.computer.org/pioneers/stibitz.html

\subsection{Collaboration}
Dans les années 1990, l'émergence des réseaux et la multiplication des
machines permettent l'émergence du Grid Computing (Informatique en Grille).

Tous ces unités centrales (PC par exemple), qui sont souvent sous
utilisées, pourraient être mobilisées pour accélérer le traitement d'une
tâche.

Le grid computing agit comme un système distribué pour le partage
collaboratif des ressources.  Celles-ci sont réparties entre
différentes unités de calcul situées sur différents sites, pays et
continents.

C’est là que le symbole du "nuage" commence à être utilisé dans les
schémas réseau pour représenter ce qui se passe "à l'extérieur" de
l'ordinateur local, dans une zone dont on ne gère pas l'infrastructure
technique mais qui collabore pour résoudre des problèmes complexes.


\subsection{As a service}

En 1999, l'entreprise Salesforce lance un logiciel de gestion de la
relation client (CRM) accessible via un simple navigateur Web.

C'est la naissance du SaaS ("Software as a Service"). C'est un
changement de paradigme commercial majeur.

On ne possède plus le logiciel, on s'y abonne et on y accède par HTTP
(Hypertext Transfer Protocol) jusque là utilisé pour transférer du
HTML (HyperText Markup Language).

\subsection{La vision commerciale}

C'est sans doute l'étape la plus concrète car elle fait vraiment
diverger le cloud d'une structure de type Grid Computing.

Dès le début des années 2000, Amazon construit une infrastructure
massive qui lui permet d'absorber des pics de trafic de sa
boutique en ligne comme par exemple à Noël.

Elle fait néanmoins le constat que ses ressources sont inutilisée 90
\% de l'année.

Le Grid Computing a été pensé avec une vision scientifique de
l'amélioration des performances. Amazon a une approche commerciale :
Louer cette infrastructure vide à d'autres entreprises.

En 2006, Amazon lance AWS (Amazon Web Services) avec des services
comme S3 (stockage) et EC2 (calcul).  Pour la première fois, une
startup pouvait louer la puissance d'un supercalculateur pour quelques
dollars par heure.


\section{Les impacts informatiques}

\subsection{Le déploiement}

Puisque l'infrastructure d'exécution est externe, il ne suffit plus de
compiler et d'exécuter, le logiciel doit être déployé sur cette infrastructure louée.

\subsection{Transparence}
Pour que l'externalisation de l'infrastructure soit transparente à l'usage, elle
devra remplir de nombreux critères parmi lesquels une haute
disponibilité, une faible latence.

Ce dernier point impacte la topologie de l'infrastructure qui tend
vers un réseau ubiquitaire se rapprochant des utilisateurs.

Plusieurs instances d'un même logiciel vont coexister dans l'infrastructure.


\subsection{Infrastructure as Code : IaC}

Traditionnellement, l'infrastructure (serveurs, câbles, pare-feu)
était du "Hardware" : quelque chose de solide qu'on touche, qu'on
visse dans une baie et qu'on configure manuellement via des câbles ou
des écrans.

Avec le cloud, "l'infrastructure devient du logiciel" (ou
Infrastructure as Code - IaC). La création et la gestion de ses
serveurs est traitée comme on traite le code d'une application.

L'administrateur système ne manipule plus des machines, mais des
abstractions. Il devient un développeur de plateforme.

Voici les 4 piliers de cette mutation :
\begin{enumerate}
\item La définition par le texte (Fichiers de configuration)

  Au lieu de cliquer sur des boutons dans une interface web pour créer
  un serveur, vous écrivez un fichier texte (souvent en format YAML ou
  JSON).

  Exemple : "Je veux 3 serveurs avec 8 Go de RAM, situés à Paris, avec
  un accès ouvert sur le port 80."

  Ce fichier est votre infrastructure.

  Pour la déployer, vous l'envoyez à une API (via des outils comme
  Terraform, CloudFormation ou Pulumi), et le Cloud "matérialise" ces
  ressources pour vous.

  \item Le versioning (Comme sur GitHub)

    Puisque votre infrastructure est maintenant un fichier texte, vous
    pouvez utiliser les mêmes outils que les développeurs :
    \begin{itemize}
    \item Historique : Vous pouvez voir qui a modifié la taille d'un serveur il y a 3 jours.
    \item Rollback : Si une modification de votre réseau provoque une
      panne, vous "revenez à la version précédente" du fichier et le
      Cloud remet l'infrastructure dans son état antérieur.
    \item Branches : Vous pouvez avoir une branche "Test" et une branche "Production" de votre infrastructure.
    \end{itemize}

  \item La reproductibilité parfaite

    C'est la fin du syndrome du "ça marche sur ma machine" ou du "le
    serveur de test n'est pas configuré comme la prod".

    Avec l'infrastructure logicielle, vous utilisez exactement le même
    script pour créer votre environnement de développement, de test et
    de production.

    Vous garantissez que les règles de sécurité et les performances
    sont identiques partout, sans risque d'erreur humaine (un oubli de
    case à cocher, par exemple).

  \item  L'immuabilité (Le concept "Phoenix")

    C'est le changement de mentalité le plus radical :

    Avant si un serveur tombait
    malade, on passait des heures à le soigner, à modifier sa
    configuration manuellement pour le réparer.

    Maintenant si un serveur a un problème ou
    doit être mis à jour, on ne le répare pas. On supprime l'instance
    et on relance le script de l'infrastructure. Le logiciel recrée un
    serveur tout neuf, parfaitement configuré, en quelques secondes.
  \end{enumerate}


  Imaginez que vous deviez déployer votre application dans 10 pays
  différents pour réduire la latence.

  Sans IaC, Il faudrait des semaines pour configurer manuellement
  chaque centre de données.

  Avec l'infrastructure logicielle : Vous lancez votre script 10 fois
  avec une variable différente pour la région. En quelques minutes,
  votre infrastructure mondiale est opérationnelle.

\subsection{Développement-Exploitation :  DEVOPS}

Le DEVOPS est une évolution de l'administration système qui adopte la
philosophie agile pour ne plus être un goulot d'étranglement.

Cette démarche essaye faire converger des contraintes antagonistes :
\begin{itemize}
\item D'un coté, l'"agile'' qui vise à accélérer les vitesses de
  développement et de mise en production.
\item De l'autre, l'administration système qui cherche à trouver une
  solution stable, fonctionnelle, sécurisée, \ldots
\end{itemize}

L'Agile est né pour les développeurs. Son but est de dire : "Arrêtons
de prévoir un projet sur 2 ans, travaillons par petits cycles pour
avoir des retours rapides du client." \ldots et donc corriger
rapidement et fréquemment le logiciel en construction.

Cette approche se focalise sur l'organisation du travail avec
notamment la mise en place de réunions régulières et fréquentes
(Scrum), l'établissement de priorités, \ldots

Or comme on l'a déjà souligné, avec une approche cloud, la mise en
production c'est le déploiement. Souvent l'Agile s'arrêtait
à la porte de la mise en production. Le développeur finissait sa tâche
"Agilement", mais le serveur n'était pas prêt.

L'administrateur système (l'Ops) n'est pas enclin au changement car le
changement apporte des pannes. Il est plutôt perçu comme un frein par les
développeurs.

Le DevOps est né parce que l'Agile ne servait à rien si le déploiement
sur les serveurs prenait 3 mois.

\subsection{Machines virtuelles et virtualisation}

La virtualisation est une technologie fondamentale qui a permis l’émergence du cloud computing. 
Elle consiste à abstraire les ressources matérielles d’un ordinateur afin de permettre l’exécution 
de plusieurs environnements isolés sur une même machine physique.

\subsubsection{Pourquoi la virtualisation ?}

Avant l’apparition de la virtualisation, un serveur physique était généralement dédié à une seule application. 
Cette approche entraînait une sous-utilisation importante des ressources matérielles (CPU, mémoire, stockage).

La virtualisation permet de résoudre ce problème en faisant coexister plusieurs machines virtuelles (VM) sur un même serveur physique.

Les principaux objectifs sont :
\begin{itemize}
\item \textbf{Optimisation des ressources} : meilleure utilisation du matériel
\item \textbf{Isolation} : chaque VM est indépendante des autres
\item \textbf{Flexibilité} : déploiement rapide de nouveaux environnements
\item \textbf{Portabilité} : une VM peut être déplacée d’un serveur à un autre
\end{itemize}

Avec pour objectif d'utiliser au mieux les ressources matérielles, d'isoler les applications, de faciliter le déploiement et la portabilité. 
%, la virtualisation est devenue un pilier du cloud computing.

\subsubsection{Historique}

La virtualisation ne date pas du cloud. Dès les années 1970, IBM développe VM/370, un système permettant à plusieurs utilisateurs dexécuter simultanément des systèmes d’exploitation sur un même ordinateur central.

Dans les années 1990, avec la démocratisation des ordinateurs personnels, des solutions comme Windows/386 permettent d’exécuter plusieurs environnements DOS.

En 1995, la machine virtuelle Java (JVM) introduit une nouvelle forme de virtualisation : une abstraction logicielle permettant d’exécuter un programme indépendamment du matériel.

Ce n’est qu’au début des années 2000 que la virtualisation devient centrale avec l’essor du cloud computing.

\subsubsection{Principe de fonctionnement}

Une machine virtuelle est une simulation logicielle d’un ordinateur physique.

Chaque VM possède : un système d’exploitation invité, de la mémoire virtuelle, un processeur virtuel et des périphériques simulés.

Ces ressources sont fournies et gérées par un logiciel appelé \textbf{hyperviseur}.

\subsubsection{Les hyperviseurs}

L’hyperviseur est la couche logicielle qui permet de créer, exécuter et gérer les machines virtuelles.

On distingue deux types :

\begin{itemize}
\item \textbf{Type 1 (bare-metal)} : fonctionne directement sur le matériel

Comme VMware ESXi ou Xen, ces hyperviseurs sont utilisés dans les datacenters et le cloud pour leur performance et leur sécurité.
Ils offrent une isolation forte entre les VM et un accès direct aux ressources matérielles, ce qui les rend adaptés aux environnements de production à grande échelle. 

\item \textbf{Type 2} : fonctionne au-dessus d’un système d’exploitation
Les hyperviseurs de type 2, sont des logiciels qui s’exécutent sur un système d’exploitation hôte, comme VirtualBox ou VMware Workstation.
Plus faciles à installer et à utiliser, ils sont souvent utilisés pour le développement, les tests ou l’apprentissage, 
mais ne sont pas adaptés aux environnements de production en raison de leur performance inférieure et de leur isolation moins robuste.
\end{itemize}

\subsubsection{Limites des machines virtuelles}

Malgré leurs avantages, les machines virtuelles présentent certaines limites :

\begin{itemize}
  \item Consommation importante de ressources (chaque VM embarque un OS complet)
  \item Temps de démarrage relativement long
  \item Complexité de gestion à grande échelle
\end{itemize}

Ces limites ont conduit à l’émergence des conteneurs, plus légers et plus rapides.

\subsubsection{Lien avec le cloud}

La virtualisation est la base du cloud computing. Elle permet aux fournisseurs comme AWS de mutualiser leurs infrastructures et de proposer des ressources à la demande.

Par exemple, une instance EC2 correspond à une machine virtuelle exécutée sur un serveur physique partagé avec d’autres utilisateurs.

Cela permet :
\begin{itemize}
\item une facturation à l’usage
\item une scalabilité rapide
\item une isolation entre clients
\end{itemize}

Cette section présente les trois principaux modèles de déploiement
cloud : \textbf{public}, \textbf{privé} et \textbf{hybride}.

Chaque modèle répond à des besoins spécifiques en termes de
flexibilité, de sécurité, de coût et de souveraineté.

\subsection{Cloud Public}

\subsubsection{Définition}
Le \textbf{cloud public} est un modèle où les ressources informatiques
(serveurs, stockage, applications) sont mises à disposition par un
fournisseur tiers via Internet.

Ces ressources sont partagées entre plusieurs clients (multi-tenancy)
et accessibles à la demande, avec une facturation généralement basée
sur l’usage (pay-as-you-go).
On le trouve chez des fournisseurs comme AWS, Microsoft Azure ou Google Cloud Platform.

\subsubsection{Avantages}

\subsection{Cloud Hybride}

\subsubsection{Définition}
Le \textbf{cloud hybride} combine les modèles public et privé,
permettant aux organisations de bénéficier des avantages des deux
approches.

Les ressources sont orchestrées pour communiquer entre elles, offrant
flexibilité et optimisation des coûts.

\subsubsection{Exemples d’architectures}
\begin{itemize}
    \item Utilisation d’un cloud privé pour les données sensibles et d’un cloud public pour les pics de charge.
    \item Intégration d’AWS Outposts (extension d’AWS sur site) avec un cloud privé OpenStack.
\end{itemize}

\subsubsection{Avantages}
\begin{itemize}
    \item \textbf{Flexibilité} : Adaptation dynamique des ressources selon les besoins.
    \item \textbf{Optimisation des coûts} : Utilisation du cloud public pour les charges variables et du privé pour les données critiques.
    \item \textbf{Continuité d’activité} : Redondance entre clouds pour une haute disponibilité.
    \item \textbf{Conformité} : Possibilité de garder les données sensibles en local tout en utilisant le cloud public pour le reste.
\end{itemize}

\subsubsection{Inconvénients}
\begin{itemize}
    \item \textbf{Complexité} : Gestion et intégration des environnements public/privé.
    \item \textbf{Sécurité} : Nécessite une stratégie unifiée pour protéger les échanges entre clouds.
    \item \textbf{Coût de mise en place} : Investissement initial pour l’interconnexion et l’orchestration.
\end{itemize}

\subsubsection{Cas d’usage typiques}
\begin{itemize}
    \item Entreprises avec des besoins variables (ex : e-commerce avec pics saisonniers).
    \item Projets nécessitant à la fois souveraineté et scalabilité (ex : santé, administration).
    \item Migration progressive vers le cloud (stratégie "lift-and-shift").
\end{itemize}

\section{Qu'est ce que le cloud}

Dans cette section, nous allons essayer de définir ce qu'est le
\textbf{cloud computing}, ses caractéristiques, et les différentes
formes qu'il peut prendre.

Il est important de comprendre les bases du cloud computing pour
pouvoir analyser les besoins et faire des choix éclairés pour notre
projet.

Il faut noter que le cloud computing est un domaine en constante
évolution, et il existe de nombreuses définitions et interprétations
différentes. Cependant, nous allons essayer de donner une définition
générale qui englobe les principales caractéristiques du cloud a
l'heure actuelle.

Le cloud computing est un \textbf{modèle de fourniture de services
  informatiques} qui permet d'accéder à des ressources informatiques
(serveurs, stockage, bases de données, réseaux, logiciels, etc.) via
Internet.

Contrairement à l’informatique "traditionnelle", où les ressources
sont hébergées localement, le cloud externalise ces infrastructures

\subsection{Q: comment faire un cloud self hosted a la render avec openstack}
Gemini a dit :
\begin{minted}{tex}
Créer un cloud "self-hosted" à la manière de **Render** ou **Heroku** (souvent appelé un **PaaS** - Platform as a Service) en utilisant **OpenStack** est un projet ambitieux.

OpenStack est complexe car c'est un **IaaS** (Infrastructure as a Service). Pour obtenir l'expérience "cliquez et déployez" de Render, vous devez construire une couche d'abstraction par-dessus OpenStack.

Voici la stratégie pour y parvenir :

-----

### 1\. La base : Déployer l'infrastructure (OpenStack)

OpenStack va gérer vos ressources brutes (calcul, stockage, réseau).

  * **La solution recommandée :** N'installez pas OpenStack "à la main". Utilisez **OpenStack-Ansible** ou **Kolla-Ansible**. Ces outils utilisent des conteneurs pour faire tourner les services Cloud, ce qui facilite grandement la maintenance.
  * **Services indispensables :**
      * *Keystone* (Identité)
      * *Nova* (Calcul/VM)
      * *Neutron* (Réseau)
      * *Glance* (Images d'OS)
      * *Cinder* (Stockage bloc)

### 2\. Le moteur de déploiement : Kubernetes sur OpenStack

Render ne déploie pas directement sur des machines virtuelles, il utilise des conteneurs. La manière la plus moderne de créer un PaaS sur OpenStack est d'utiliser OpenStack pour créer des clusters Kubernetes.

  * **OpenStack Magnum :** C'est le service OpenStack dédié à la création de clusters de conteneurs (Kubernetes).
  * **Pourquoi ?** Kubernetes gérera l'orchestration de vos applications, tandis qu'OpenStack gérera les serveurs physiques en dessous.

### 3\. La couche "Render-like" (Le PaaS)

Pour avoir l'expérience utilisateur de Render (Git Push → Déploiement), vous devez installer une plateforme de développement sur votre Kubernetes. Voici les meilleures options open-source :

  * **Coolify :** Probablement l'alternative la plus proche de Render aujourd'hui. Très simple à auto-héberger.
  * **Dokku :** Un "mini-Heroku" très puissant, parfait si vous n'avez qu'un seul gros serveur.
  * **CapRover :** Très visuel, avec un "App Store" pour déployer des bases de données en un clic.

### 4\. Automatiser le flux (CI/CD)

Pour que cela ressemble à Render, le processus doit être automatique :

1.  **Git :** Vous poussez votre code sur votre propre instance **GitLab** ou **Gitea**.
2.  **Build :** Un outil comme **Cloud Native Buildpacks** (utilisé par Render) transforme votre code en image de conteneur sans que vous ayez à écrire de Dockerfile.
3.  **Run :** L'image est poussée sur votre cluster géré par OpenStack.

-----

### Résumé de la pile technologique (Stack)

| Couche | Technologie recommandée |
| :--- | :--- |
| **Matériel** | Vos serveurs physiques |
| **Infrastructure (IaaS)** | **OpenStack** (via Kolla-Ansible) |
| **Orchestration** | **Kubernetes** (via OpenStack Magnum) |
| **Interface PaaS** | **Coolify** ou **Dokku** |
| **Accès Public** | **Traefik** ou **Nginx Ingress** (pour les URL en [https://www.google.com/search?q=.votre-cloud.com](https://www.google.com/search?q=.votre-cloud.com)) |

### Est-ce une bonne idée ?

  * **Les avantages :** Contrôle total de vos données, aucun coût de bande passante caché, apprentissage immense.
  * **Les inconvénients :** **Énorme charge de maintenance.** OpenStack est un "monstre" qui nécessite beaucoup de RAM (minimum 16-32 Go juste pour le plan de contrôle) et une surveillance constante.

Si votre objectif est simplement d'avoir un "Render à la maison" sans la complexité d'OpenStack, installer **Coolify** directement sur un serveur Linux (Ubuntu) est 10 fois plus simple.

Si tu veux une expérience **"à la Render"** (Git Push, déploiement automatique, bases de données en un clic) sans t'arracher les cheveux sur OpenStack, **Coolify** est l'outil parfait. C'est ce qu'on appelle un "PaaS auto-hébergé".

Voici comment transformer un simple serveur en ton propre Cloud privé avec Coolify.

---

### 1. La philosophie Coolify
Contrairement à OpenStack qui gère des machines virtuelles, Coolify gère des **conteneurs (Docker)**. Il s'installe sur une machine Linux vierge et devient ton centre de contrôle.

### 2. Installation (La minute magique)
Il te suffit d'une machine sous Ubuntu (ou Debian/CentOS) avec au moins **2 Go de RAM**. Connecte-toi en SSH et lance cette commande :

```bash
curl -fsSL https://get.coollabs.io/coolify/install.sh | bash
```

Une fois terminé, tu accèdes à une interface web ultra-propre (généralement sur le port `8000`).

### 3. Comment il reproduit l'expérience "Render"

Coolify automatise tout ce qui est normalement pénible dans l'administration système :

*   **Connexion Git :** Tu lies ton compte GitHub ou GitLab. Dès que tu pousses du code, Coolify le détecte, crée une nouvelle image de ton app et la déploie.
*   **Reverse Proxy intégré :** Il installe automatiquement **Traefik**. Tu n'as pas à configurer de fichiers Nginx compliqués pour avoir tes noms de domaine.
*   **SSL Automatique :** Il génère tes certificats **Let's Encrypt** (HTTPS) tout seul.
*   **Bases de données :** Tu cliques sur "PostgreSQL" ou "Redis", et il te déploie une instance sécurisée avec une interface de gestion intégrée.

---

### 4. La configuration "Multi-Serveurs" (Le vrai Cloud)
Si tu veux vraiment faire comme Render et ne pas tout mettre sur une seule machine :
1.  Tu installes Coolify sur un serveur principal (le **Control Plane**).
2.  Tu achètes d'autres serveurs vides (tes **Nodes**).
3.  Dans l'interface Coolify, tu ajoutes l'IP de tes autres serveurs via SSH.
4.  Désormais, quand tu déploies une application, tu peux choisir sur quel serveur elle doit tourner.

### Le conseil du "pro"
Si tu veux un Cloud robuste à la maison ou en entreprise :
