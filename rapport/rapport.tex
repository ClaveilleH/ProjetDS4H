\section{Qu'est ce que le cloud}

Dans cette section, nous allons essayer de définir ce qu'est le \textbf{cloud computing}, ses caractéristiques, et les différentes formes qu'il peut prendre.
Il est important de comprendre les bases du cloud computing pour pouvoir analyser les besoins et faire des choix éclairés pour notre projet.
Il faut noter que le cloud computing est un domaine en constante évolution, et il existe de nombreuses définitions et interprétations différentes.
Cependant, nous allons essayer de donner une définition générale qui englobe les principales caractéristiques du cloud a l'heure actuelle.


Le cloud computing est un \textbf{modèle de fourniture de services informatiques} qui permet d'accéder à des ressources informatiques (serveurs, stockage, bases de données, réseaux, logiciels, etc.) via Internet. 
Contrairement à l’informatique traditionnelle, où les ressources sont hébergées localement, le cloud externalise ces infrastructures vers des \textbf{centres de données distants}, gérés par des prestataires spécialisés.
\cite{ISO2026,NIST2026}.


\subsection{Caractéristiques clés (NIST/ISO, 2026)}
\begin{itemize}
    \item \textbf{Accès réseau omniprésent} : Les ressources sont accessibles depuis n’importe quel appareil connecté à Internet.
    \item \textbf{Pool de ressources mutualisées} : Les infrastructures sont partagées et allouées dynamiquement selon les besoins (virtualisation, conteneurs).
    \item \textbf{Élasticité et scalabilité} : Les ressources peuvent être ajustées automatiquement, en temps réel, pour répondre à la demande.
    \item \textbf{Self-service à la demande} : L’utilisateur peut provisionner des ressources (ex : machines virtuelles, bases de données) sans intervention humaine du fournisseur.
    \item \textbf{Facturation à l’usage} : Modèle économique "pay-as-you-go", où le coût est proportionnel à la consommation réelle.
\end{itemize}

\subsection{Évolution récente (2026)}
En 2026, le cloud n’est plus seulement un \textbf{lieu de stockage} ou d’hébergement, mais une \textbf{plateforme d’innovation} intégrant :
\begin{itemize}
    \item L’\textbf{intelligence artificielle} (ex : GPU-as-a-Service pour l’entraînement de modèles, AIOps pour l’optimisation automatique des ressources) \cite{LeBigData2026}.
    \item L’\textbf{Edge Computing} : traitement des données à la périphérie du réseau pour réduire la latence, notamment pour l’IoT et les applications temps réel \cite{Syslearn2026}.
    \item Les \textbf{architectures hybrides et multi-cloud} : les entreprises combinent clouds publics, privés et infrastructures locales pour éviter le \textit{vendor lock-in} et optimiser flexibilité/sécurité \cite{ObjetConnecte2026}.
    \item La \textbf{souveraineté des données} : développement de clouds locaux (ex : OVHcloud en France) pour se conformer aux réglementations comme le RGPD \cite{TechInsider2026}.
\end{itemize}

\subsection{Pourquoi cette définition est-elle importante pour notre projet ?}
Cette compréhension permet d’identifier :
\begin{itemize}
    \item Les \textbf{avantages} du cloud (flexibilité, coût, innovation) mais aussi ses \textbf{limites} (dépendance, sécurité, latence).
    \item Les \textbf{critères de choix} entre cloud public, privé, hybride ou \textit{self-hosted}, en fonction des besoins en souveraineté, performance et budget.
\end{itemize}

\section{Différentes méthodes}

Cette section présente les trois principaux modèles de déploiement cloud : \textbf{public}, \textbf{privé} et \textbf{hybride}. Chaque modèle répond à des besoins spécifiques en termes de flexibilité, de sécurité, de coût et de souveraineté.

\subsection{Cloud Public}

\subsubsection{Définition}
Le \textbf{cloud public} est un modèle où les ressources informatiques (serveurs, stockage, applications) sont mises à disposition par un fournisseur tiers via Internet. Ces ressources sont partagées entre plusieurs clients (multi-tenancy) et accessibles à la demande, avec une facturation généralement basée sur l’usage (pay-as-you-go).

\subsubsection{Exemples de fournisseurs}
\begin{itemize}
    \item Amazon Web Services (AWS)
    \item Microsoft Azure
    \item Google Cloud Platform (GCP)
    \item IBM Cloud
\end{itemize}

\subsubsection{Avantages}
\begin{itemize}
    \item \textbf{Coût réduit} : Pas d’investissement initial en infrastructure, paiement à l’usage.
    \item \textbf{Élasticité} : Scalabilité automatique pour répondre à la demande.
    \item \textbf{Maintenance externalisée} : Le fournisseur gère les mises à jour, la sécurité physique et la disponibilité.
    \item \textbf{Accès global} : Disponibilité mondiale avec des data centers répartis géographiquement.
\end{itemize}

\subsubsection{Inconvénients}
\begin{itemize}
    \item \textbf{Dépendance au fournisseur} : Risque de \textit{vendor lock-in} et dépendance aux conditions tarifaires.
    \item \textbf{Sécurité et conformité} : Les données sont hébergées chez un tiers, ce qui peut poser des problèmes de conformité (ex : RGPD).
    \item \textbf{Latence} : Peut être élevée selon la localisation des data centers.
    \item \textbf{Manque de contrôle} : L’utilisateur n’a pas la main sur l’infrastructure sous-jacente.
\end{itemize}

\subsubsection{Cas d’usage typiques}
\begin{itemize}
    \item Hébergement de sites web et applications grand public.
    \item Déploiement de services SaaS (ex : Gmail, Salesforce).
    \item Environnements de développement/test éphémères.
\end{itemize}

\subsection{Cloud Privé}

\subsubsection{Définition}
Le \textbf{cloud privé} est une infrastructure cloud dédiée à une seule organisation. Il peut être hébergé en interne (on-premise) ou chez un prestataire externe, mais reste exclusif à l’organisation. Ce modèle offre un contrôle total sur les ressources et la sécurité.

\subsubsection{Exemples de solutions}
\begin{itemize}
    \item OpenStack (open source)
    \item VMware Cloud
    \item Microsoft Azure Stack
    \item Clouds souverains (ex : OVHcloud, 3DS Outscale)
\end{itemize}

\subsubsection{Avantages}
\begin{itemize}
    \item \textbf{Sécurité renforcée} : Isolation des données et contrôle total sur l’infrastructure.
    \item \textbf{Conformité} : Idéal pour les secteurs réglementés (santé, finance, défense).
    \item \textbf{Personnalisation} : Adaptation fine des ressources aux besoins spécifiques.
    \item \textbf{Performances} : Latence réduite et ressources dédiées.
\end{itemize}

\subsubsection{Inconvénients}
\begin{itemize}
    \item \textbf{Coût élevé} : Investissement initial important en matériel et maintenance.
    \item \textbf{Complexité de gestion} : Nécessite des compétences internes pour l’administration.
    \item \textbf{Scalabilité limitée} : L’extension des ressources peut être lente et coûteuse.
    \item \textbf{Risque de sous-utilisation} : Les ressources dédiées peuvent être sous-exploitées, entraînant un gaspillage.
    \item \textbf{Manque de redondance} : En cas de panne, la récupération peut être plus complexe sans les infrastructures de secours d’un cloud public.
\end{itemize}

\subsubsection{Cas d’usage typiques}
\begin{itemize}
    \item Environnements sensibles (données médicales, financières).
    \item Applications critiques nécessitant une haute disponibilité.
    \item Projets nécessitant une souveraineté totale sur les données.
\end{itemize}

\subsection{Cloud Hybride}

\subsubsection{Définition}
Le \textbf{cloud hybride} combine les modèles public et privé, permettant aux organisations de bénéficier des avantages des deux approches. Les ressources sont orchestrées pour communiquer entre elles, offrant flexibilité et optimisation des coûts.

\subsubsection{Exemples d’architectures}
\begin{itemize}
    \item Utilisation d’un cloud privé pour les données sensibles et d’un cloud public pour les pics de charge.
    \item Intégration d’AWS Outposts (extension d’AWS sur site) avec un cloud privé OpenStack.
\end{itemize}

\subsubsection{Avantages}
\begin{itemize}
    \item \textbf{Flexibilité} : Adaptation dynamique des ressources selon les besoins.
    \item \textbf{Optimisation des coûts} : Utilisation du cloud public pour les charges variables et du privé pour les données critiques.
    \item \textbf{Continuité d’activité} : Redondance entre clouds pour une haute disponibilité.
    \item \textbf{Conformité} : Possibilité de garder les données sensibles en local tout en utilisant le cloud public pour le reste.
\end{itemize}

\subsubsection{Inconvénients}
\begin{itemize}
    \item \textbf{Complexité} : Gestion et intégration des environnements public/privé.
    \item \textbf{Sécurité} : Nécessite une stratégie unifiée pour protéger les échanges entre clouds.
    \item \textbf{Coût de mise en place} : Investissement initial pour l’interconnexion et l’orchestration.
\end{itemize}

\subsubsection{Cas d’usage typiques}
\begin{itemize}
    \item Entreprises avec des besoins variables (ex : e-commerce avec pics saisonniers).
    \item Projets nécessitant à la fois souveraineté et scalabilité (ex : santé, administration).
    \item Migration progressive vers le cloud (stratégie "lift-and-shift").
\end{itemize}


\section{Contexte d'utilisation} % comment moi je vais l'utiliser
Dans le cadre du projet \textbf{DS4H}, il m'a été confié la mission de concevoir et déployer une \textbf{solution self-hosted} 
pour remplacer le service \textbf{Render} utilisé dans le cours d’\textbf{IoT}. L’objectif est de fournir une alternative locale, 
souveraine et maîtrisée, permettant aux étudiants et enseignants de déployer, 
gérer et monitorer des applications IoT sans dépendre d’un service cloud externe.

Cette initiative s’inscrit dans une démarche visant à :
\begin{itemize}
    \item \textbf{Réduire la dépendance} aux services tiers, en alignement avec les principes de souveraineté numérique et de résilience pédagogique.
    \item \textbf{Reproduire les fonctionnalités essentielles} de Render (déploiement continu, gestion des environnements, monitoring basique) 
    tout en s’adaptant aux besoins spécifiques du cours (protocoles IoT, gestion des devices, etc.).
    \item \textbf{Faciliter l’apprentissage} en offrant un environnement contrôlé et accessible, 
    propice à l’expérimentation des concepts d’IoT et d’infrastructure.
\end{itemize}

L’absence de contraintes techniques précises laisse une grande flexibilité pour le choix de la solution, qui sera déterminé lors de l’\textbf{analyse des besoins} et des choix technologiques présentés dans les sections suivantes.
\section{Analyse des besoins}
Le but de ce projet étant d'implémenter une solution de cloud self-hosted pour remplacer Render, il est essentiel de comprendre les besoins spécifiques liés à ce contexte d’utilisation.
\begin{itemize}
    \item \textbf{Deploiement facile} : Les utilisateurs doivent pouvoir déployer leurs code sans que la complexité technique ne soit un obstacle.
    \item \textbf{Gestion des comptes utilisateurs} : Il doit être possible de créer et gérer des comptes pour les étudiants.
    \item \textbf{Monitoring basique} : Les utilisateurs doivent pouvoir surveiller l’état de leurs applications (logs, ressources utilisées).
\end{itemize}
